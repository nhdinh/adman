  <behavior>
    - `Start-AdmanUserOffboarding -Identity 'jdoe'` disables the user, removes non-protected group memberships, moves to quarantine OU, and writes a Success audit with `OriginalOU` and `Groups`.
    - If the quarantine OU is outside managed scope, the workflow throws before any AD write.
    - If a step fails, later steps are skipped and a Failure audit is written.
  </behavior>
  <action>
    Create Public/Start-AdmanUserOffboarding.ps1:
      - Declare `[CmdletBinding(SupportsShouldProcess, ConfirmImpact='High')]` with mandatory `Identity` and optional `Force`.
      - Throw the standard WR-01 init message if `$script:Config.ManagedOUs` is missing.
      - Read `$quarantineOu = $script:Config.templates.offboarding.quarantineOU`; validate it is under a managed-OU root using `ConvertTo-AdmanNormalizedDn` (same boundary-anchored check as Move-AdmanUser).
      - Resolve the user via `Resolve-AdmanTarget` to get `DistinguishedName` and `memberOf`.
      - Compute `$originalOu` by stripping the first RDN from the user's DN.
      - Compute `$groupsToRemove` = `@($user.memberOf | Where-Object { $_ -notin @($script:ProtectedGroupDns) })`. Protected groups are those whose DN appears in `$script:ProtectedGroupDns`.
      - Wrap the workflow in try/catch and execute:
        1. `Disable-AdmanUser -Identity $Identity -Force:$Force -WhatIf:$WhatIfPreference`
        2. For each `$g` in `$groupsToRemove`: `Remove-AdmanGroupMember -Identity $Identity -GroupIdentity $g -Force:$Force -WhatIf:$WhatIfPreference`
        3. `Move-AdmanUser -Identity $Identity -TargetPath $quarantineOu -Force:$Force -WhatIf:$WhatIfPreference`
        4. `Write-AdmanAudit -Verb 'Start-AdmanUserOffboarding' -Target $user -Result 'Success' -OriginalOU $originalOu -Groups $groupsToRemove -WhatIf:$WhatIfPreference`
      - On catch: write `Write-AdmanAudit -Verb 'Start-AdmanUserOffboarding' -Target $user -Result 'Failure' -Reason $_.Exception.Message -WhatIf:$WhatIfPreference`, then rethrow.
      - After the try block (or in a Finally block if not under -WhatIf), write a plain-text checklist to the console listing typical cleanup items (mailbox, home directory, GPO) with explicit "manual only" wording; do NOT automate any of them.
    Create tests/Workflow.Offboarding.Tests.ps1 using the standard harness. Mock the composed verbs and audit writer. Cover happy path, quarantine-OU scope refusal, protected-group preservation, Failure audit on step throw, and checklist output.

|----------|-------------|
| Audit log -> restore logic | Authoritative restore state crosses from persistent log into workflow |
| Operator -> offboarding workflow | Identity selection and quarantine OU configuration |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-09 | Tampering | Restore uses a stale or tampered offboarding record | high | mitigate | Read only `Success` records from the synchronous fail-closed audit log; validate the target is still in the configured quarantine OU before reversing. |
| T-04-10 | Denial of service / data loss | Offboarding cleanup automation deletes mailbox/home dir | medium | mitigate | Cleanup is rendered as a plain-text checklist only; no automated deletion (FLOW-02). |
| T-04-11 | Elevation of privilege | Restore moves an account outside managed scope | medium | mitigate | Re-check `state.OriginalOU` against managed-OU roots before calling `Move-AdmanUser`. |
| T-04-12 | Repudiation | Offboarding partial failure leaves no Failure audit | medium | mitigate | try/catch writes a Failure audit record before rethrowing (FLOW-04). |
</threat_model>
