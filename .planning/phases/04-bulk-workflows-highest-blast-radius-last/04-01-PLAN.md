## Review Findings Addressed

### Review finding: Bulk typed-count confirmation does not run for small filtered sets
**Disposition:** Implemented. Extended `Confirm-AdmanAction` with `-RequireTypedCount` (Task 2) and the bulk engine always invokes it (Task 3). This satisfies the BULK-02 requirement for typed confirmation of the exact filtered count at any size.

### Review finding: Risk of N additional inner confirmations after outer bulk confirmation
**Disposition:** Implemented. After the outer confirmation the engine calls `Invoke-AdmanMutation` per item with `-Force:$true` so the inner gate still runs policy/audit but skips its own confirmation prompt (Task 3). Tests verify `Confirm-AdmanAction` is invoked once per job.

### Review finding: Group destination policy can fail after confirmation
**Disposition:** Implemented. For `AddGroup`/`RemoveGroup` the engine resolves and validates every distinct `GroupIdentity` via `Resolve-AdmanGroup` + `Test-AdmanGroupAllowed` before cap/confirm (Task 3).

### Review finding: CSV validation did not explicitly reject missing required headers
**Disposition:** Implemented. `Import-AdmanBulkCsv` now rejects unknown, duplicate, and missing required headers (`Action`, `Identity`) before returning rows (Task 2).

### Review finding: UPN source `$script:Config.Domain` does not exist in config schema/defaults
**Disposition:** Implemented in this plan. Added a top-level `domain` key to `config/adman.schema.json` and `config/adman.defaults.json` (Task 1); onboarding will consume it in plan 04-02.

### Review finding: Adding `domain` to schema `required` breaks existing installed configs without loader migration
**Disposition:** Implemented. Extended `Initialize-AdmanConfig` with a Phase 4 additive-defaults migration block that seeds missing `domain` and `templates` from shipped defaults before validation, preserving any existing user values (Task 1). `tests/Config.Load.Tests.ps1` covers migration.

### Review finding: Bulk pre-filtering plus per-item gate calls resolve/policy-check targets twice, risking stale blast radius
**Disposition:** Mitigated and documented. The outer filter produces the confirmed set and cap; the inner `Invoke-AdmanMutation` re-check remains authoritative. Tests assert that a target changing status between outer filter and inner execution is caught and reported by the inner gate (Task 3).

### Review finding: `-Force` can bypass typed-count confirmation on direct invocation
**Disposition:** Accepted and documented. `Invoke-AdmanBulkAction -Force` skips the outer typed-count confirmation while still running per-item policy and audit; this is an intentional senior escape hatch, not the TUI path. Comment-based help and tests document the behavior (Task 3).
