  <action>
    Create Private/Utility/Escape-AdmanAdFilterLiteral.ps1 with a function Escape-AdmanAdFilterLiteral that takes [Parameter(Mandatory)][AllowEmptyString()][string]$Value and returns a string safe for interpolation into an AD PowerShell -Filter string-literal position (between single quotes).
    Escaping rules (HIGH-1 — distinct from RFC4515):
    - Single quote (') is DOUBLED ('') — this is the AD -Filter string-literal escape. A name like O'Brien becomes O''Brien and the filter still parses as a single literal.
    - Backslash (\) is DOUBLED (\\) — backslash is the AD -Filter escape character; a literal backslash must be escaped.
    - Wildcard characters (* and ?) are NOT escaped by this helper. The Find verbs intentionally use -like semantics on -Name (D-02); callers that need exact-match semantics use -eq and the caller is responsible for not passing user-controlled wildcards into -eq positions. Document this in comment-based help.
    - Parentheses ( ) and NUL are NOT special inside an AD -Filter string literal (unlike RFC4515 LDAP assertions) and are passed through unchanged.
    Return the empty string for $null or empty input.
    The function name MUST be Escape-AdmanAdFilterLiteral (NOT Escape-AdmanLdapFilterValue) so call sites cannot accidentally confuse the two helpers. Comment-based help MUST state: "Use this helper for AD PowerShell -Filter strings. Use Escape-AdmanLdapFilterValue for -LDAPFilter (RFC4515) strings. The two are NOT interchangeable."
    Create tests/Utility.EscapeFilter.Tests.ps1 with Pester 6 unit tests covering at minimum:
    - Single quote: "O'Brien" -> "O''Brien"
    - Backslash: "CN=Doe\John" -> "CN=Doe\\John"
    - Combined: "O'\Brien" -> "O''\\Brien"
    - Pass-through: "normal" -> "normal"; "O(Brien)" -> "O(Brien)"; "O*Brien" -> "O*Brien" (wildcard preserved for -like)
    - Empty/null: '' -> ''; $null -> ''
    - Assertion that the helper does NOT invoke Escape-AdmanLdapFilterValue (the two are independent).
