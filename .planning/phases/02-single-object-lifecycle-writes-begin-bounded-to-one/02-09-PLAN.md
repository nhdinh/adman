<objective>
Stop menu crashes on non-DN input by introducing a shared identity/DN resolver (`Resolve-AdmanIdentity`) and wiring it into the menu prompt engine with re-prompt-on-failure. UAT Test 2 showed entering `adman-test` (not a DN) at the parent-OU prompt crashed at `Invoke-AdmanMutation.ps1:90` with a raw `Get-ADObject` error. UAT Test 4 showed entering `uat-reset1` (sAMAccountName) at the identity prompt crashed the app. Both are the same defect class: no prompt-time validation/resolution of identity input.

Purpose: The guided TUI must catch malformed input at prompt time and re-prompt, not crash deep in the gate. This is a BLOCKER for the menu path (the core UX for the junior admin persona).

Output: New `Private/Safety/Resolve-AdmanIdentity.ps1` resolver. `Read-AdmanActionParams.ps1` free-text branch gains a `Type` dispatch for `AdIdentity` and `AdOuDn`. `Get-AdmanMenuDefinition.ps1` PromptSpec entries updated with the new Type values. New Pester test proves resolution + re-prompt behavior.
</objective>

    Create `Private/Safety/Resolve-AdmanIdentity.ps1` with the standard file header (`#Requires -Version 5.1`, doc block referencing G-02-2/G-02-4, `Set-StrictMode -Version Latest`).

    Export ONE function `Resolve-AdmanIdentity` with this contract:

      param(
          [Parameter(Mandatory)][string]$InputValue,
          [Parameter(Mandatory)][ValidateSet('AdObject','AdOuDn')][string]$Kind
      )

    Behavior for `Kind='AdObject'` (identity prompts — user/computer/group member):
      1. If `$InputValue` looks like a DN (contains '=' AND ','), pass it through to `Get-ADObject -Identity $InputValue -Server $script:Config.DC -Properties objectSid, objectClass, DistinguishedName, memberOf -ErrorAction Stop`. On success return the ADObject. On throw, re-throw a typed error: `"Cannot resolve identity '$InputValue' to an AD object: $($_.Exception.Message)"`.
      2. Otherwise treat `$InputValue` as a sAMAccountName: escape via `Escape-AdmanAdFilterLiteral`, then `Get-ADObject -Filter "sAMAccountName -eq '$esc'" -Server $script:Config.DC -Properties objectSid, objectClass, DistinguishedName, memberOf -ErrorAction Stop`. If exactly one hit, return it. If zero hits, throw `"No AD object found with sAMAccountName '$InputValue'."`. If multiple hits (should not happen — sAMAccountName is forest-unique), throw `"Multiple AD objects match sAMAccountName '$InputValue'."`.

    Behavior for `Kind='AdOuDn'` (parent-OU and destination-OU prompts):
      1. Validate the input LOOKS like a DN (contains '=' AND ','). If not, throw `"'$InputValue' is not a distinguished name. Enter the full OU DN (e.g. OU=adman-test,DC=lab,DC=local)."`
      2. Attempt `Get-ADOrganizationalUnit -Identity $InputValue -Server $script:Config.DC -Properties DistinguishedName -ErrorAction Stop`. On success return the OU object. On throw, re-throw: `"Cannot resolve OU '$InputValue': $($_.Exception.Message)"`.

    The function does NO scope checking (that is Test-AdmanTargetAllowed's job downstream). It only resolves and validates existence.

    `-Server $script:Config.DC` pinning on every AD cmdlet call (per the established pattern).

    Do NOT export this function from the module manifest — it is Private/ and consumed only by Read-AdmanActionParams.
  </action>
