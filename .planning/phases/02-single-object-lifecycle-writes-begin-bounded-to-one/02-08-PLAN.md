<objective>
Unblock the create flow (USER-02) by making the AD-target branch of `Write-AdmanAudit` tolerant of a null `objectSid`. For create-verbs the audit target is fabricated by `Resolve-AdmanCreateTarget` BEFORE the AD object exists, so `objectSid` is `$null`. Under `Set-StrictMode -Version Latest`, the expression `($t.objectSid.Value)` at `Write-AdmanAudit.ps1:78` throws "The property 'Value' cannot be found on this object", the PENDING write fails, and the fail-closed catch refuses `New-ADUser`. Verbs against existing objects pass a real `SecurityIdentifier` and are unaffected.

Purpose: This is a BLOCKER. UAT Test 3 showed New-AdmanUser dead — the fail-closed audit writer (correctly) refused the mutation because the audit record itself could not be constructed. The fix is a defensive null/type check, NOT a relaxation of fail-closed.

Output: `Write-AdmanAudit.ps1` AD-target branch uses guarded SID extraction (mirroring the local-target branch's existing guarded `$t.SID` handling at line 87). New Pester test proves the synthetic-target PENDING write succeeds under StrictMode and the fail-closed property is intact.
</objective>

          $t.objectSid.Value

          [string]$t.objectSid
