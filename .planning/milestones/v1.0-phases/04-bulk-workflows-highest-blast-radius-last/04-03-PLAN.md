      - Throw the standard WR-01 init message if `$script:Config.ManagedOUs` is missing.
      - Read `$quarantineOu = $script:Config.templates.offboarding.quarantineOU`; validate it is under a managed-OU root using `ConvertTo-AdmanNormalizedDn` (same boundary-anchored check as Move-AdmanUser).
      - Resolve the user via `Resolve-AdmanTarget -Targets @($Identity)` to get `DistinguishedName`, `objectSid`, and `memberOf`.
      - Compute `$originalOu` by stripping the first RDN from the user's DN.
      - Compute protected groups by resolving each DN in `$user.memberOf` via `Resolve-AdmanGroup` where possible. A group is protected if any of the following is true:

      - Call `Confirm-AdmanAction -Verb 'Start-AdmanUserOffboarding' -Targets @($user) -Force:$Force` once before any destructive step; throw if Declined.

      - On catch: write `Write-AdmanAudit -Verb 'Start-AdmanUserOffboarding' -Target $user -Result 'Failure' -Reason $_.Exception.Message -WhatIf:$WhatIfPreference`, then rethrow.
      - After the try block (or in a Finally block if not under -WhatIf), write a plain-text checklist to the console listing typical cleanup items (mailbox, home directory, GPO) with explicit "manual only" wording; do NOT automate any of them.

## Review Findings Addressed

### Review finding: Restore state lookup uses substring matching against pipe-joined target
**Disposition:** Implemented. `Get-AdmanOffboardingState` resolves the user first, then matches audit records by exact `targets[].dn` or `targets[].sid` equality (Task 3).

### Review finding: Restore state lookup includes `whatIf=true` records
**Disposition:** Implemented. `Get-AdmanOffboardingState` filters to records where `result -eq 'Success'` and `whatIf -eq $false` (Task 3). Tests verify dry-run offboarding records are excluded.

### Review finding: 30-day audit lookback cutoff is not a stated product decision
**Disposition:** Implemented. Removed the 30-day limit; `Get-AdmanOffboardingState` enumerates all `audit-*.jsonl` files in the audit directory (Task 3).

### Review finding: Protected-group classification uses DN-only comparison and misses unresolved SIDs
**Disposition:** Implemented. Offboarding resolves each `memberOf` group and classifies it as protected if its SID, RID, or DN appears in the protected sets (`ProtectedSIDs`, `DenyRids`, `ProtectedGroupDns`), including SID strings stored in `ProtectedGroupDns` (Task 2).

### Review finding: Offboarding workflow lacks an outer confirmation before forced inner destructive verbs
**Disposition:** Implemented. Added a single `Confirm-AdmanAction -Verb 'Start-AdmanUserOffboarding'` call before any destructive step; inner `Disable-AdmanUser`, `Remove-AdmanGroupMember`, and `Move-AdmanUser` calls use `-Force:$true` only after confirmation (Task 2). Tests verify confirmation occurs exactly once and inner verbs are forced only after it.

### Review finding: Restore enables the account before re-adding groups and moving back to the original OU
**Disposition:** Implemented. Reordered restore to add groups, move back to the original OU, and enable the account last, so a partial restore leaves the account disabled (Task 3). Tests verify the ordering invariant.

### Review finding: Audit schema tests may need updated exact-key expectations for offboarding records
**Disposition:** Implemented. `Write-AdmanAudit` only emits `originalOU` and `groups` when supplied, preserving the exact-key invariant for all other records. Acceptance criteria require `tests/Audit.Schema.Tests.ps1` to assert both the absence of these keys on non-offboarding records and their presence on offboarding records (Task 1).
